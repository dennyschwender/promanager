# Email Templates & Magic Login Links Design

**Date:** 2026-04-15

## Context

ProManager currently sends emails using a mix of inline HTML strings (welcome, reset) and locale JSON strings (reminder, attendance). None of the emails include a link back to the app, and there are no proper HTML templates. This spec adds:

1. Proper Jinja2 HTML email templates for all 5 email types
2. A stateless magic login link that auto-authenticates the recipient and lands them on the relevant page

## Magic Link System

### Token generation

A new `create_magic_link(user_id, redirect_path)` function in `services/auth_service.py` uses the existing `itsdangerous.TimestampSigner` (same key as session cookies) to sign a `{user_id}:{redirect_path}` payload with a 48-hour expiry. Returns a full URL: `{APP_URL}/auth/magic?token=<signed>`.

A matching `verify_magic_link(token)` function unsigns the token and returns `(user_id, redirect_path)` or raises on invalid/expired.

### Magic link route

New `GET /auth/magic?token=<token>` route in `routes/auth.py`:
- Calls `verify_magic_link(token)`
- On success: sets the standard `session_user_id` cookie (reusing `create_session_cookie`) and redirects to the decoded path
- On failure (expired, invalid): redirects to `/auth/login` with a flash message

### APP_URL config

New `APP_URL: str = "http://localhost:7000"` setting in `app/config.py`. Set via `.env`:

```
APP_URL=https://promanager.mennylenderr.ch
```

**If `APP_URL` is the default localhost value, the magic link button is omitted from emails** ŌĆö the email still sends normally, just without the login button. This prevents broken links in dev/test and avoids hard failures on misconfiguration.

### Redirect targets per email type

| Email type | Button destination |
|---|---|
| Reminder | `/events/{event_id}` |
| Attendance | `/events/{event_id}` |
| Welcome | `/dashboard` |
| Reset | `/dashboard` |
| Notification | `/notifications` |

## Email Templates

### Template structure

All templates live in `templates/email/` and use Jinja2, the same engine used for web pages.

```
templates/email/
  base.html           ŌåÉ shared layout
  reminder.html
  attendance.html
  welcome.html
  reset.html
  notification.html
```

### base.html layout

Minimal, email-client-safe HTML:
- Header: `APP_NAME` in bold
- Content block (`{% block content %}`)
- Optional magic link button: "Open in ProManager ŌåÆ" ŌĆö only rendered when `magic_link` is not None
- Footer: app name + "You received this email because you are registered in {app_name}."

Button styled with inline CSS (no external stylesheets ŌĆö required for email clients).

### Template content

**reminder.html** ŌĆö Hi {name}, reminder for upcoming event. Table: Event / When / Location. Button ŌåÆ event page.

**attendance.html** ŌĆö Hi {name}, please confirm your attendance for {event_name} on {date}. Button ŌåÆ event page.

**welcome.html** ŌĆö Your account has been created. Username: {username}. Temporary password: {password}. Button ŌåÆ dashboard.

**reset.html** ŌĆö Your password has been reset. New credentials: Username: {username} / Password: {password}. Button ŌåÆ dashboard.

**notification.html** ŌĆö {title}. {body}. Button ŌåÆ `/notifications`.

### Plain-text fallback

Plain text is auto-generated by stripping HTML tags from the rendered template output. No separate locale text strings needed.

### Template rendering

New `render_email_template(name, context)` helper in `services/email_service.py` uses FastAPI's existing Jinja2 `Environment` to render `templates/email/{name}.html`. Context always includes `app_name`, `magic_link` (or `None`), and template-specific variables.

## Updated email_service.py

All send functions are updated to use templates. Two new functions replace inline route code:

```python
def send_welcome_email(to, username, password, locale, magic_link) -> bool
def send_reset_email(to, username, password, locale, magic_link) -> bool
```

Existing `send_event_reminder()` and `send_attendance_request()` gain a `magic_link` parameter.

A new `send_notification_email()` function is added for the notification channel.

All functions call `create_magic_link(user_id, redirect_path)` internally ŌĆö callers pass `user_id` and the service builds the link.

## routes/users.py

Replace inline `send_email(...)` calls with `send_welcome_email()` and `send_reset_email()`.

## Locale changes

New keys added to all 4 locale files (`en`, `it`, `fr`, `de`) in the `email` section:

```
email.login_button       "Open in ProManager ŌåÆ"
email.footer             "You received this email because you are registered in %{app_name}."
email.reminder_subject   (kept ŌĆö already exists)
email.attendance_subject (kept ŌĆö already exists)
email.welcome_subject    "Your %{app_name} account"
email.reset_subject      "Your %{app_name} password has been reset"
email.notification_subject "%{title}"
```

Existing `email.reminder_body`, `email.reminder_body_html`, `email.attendance_body`, `email.attendance_body_html` locale keys are retired (templates replace them). `users.email_subject` is replaced by `email.welcome_subject` and `email.reset_subject`.

## test_email.py updates

Pass `--event-id` and `--user-id` through to the send functions so test emails include a real magic link when `APP_URL` is configured.
