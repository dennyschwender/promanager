# Bug Fixes — 2026-05-28

## 1. Action dropdown (⋯) broken on Members, Events, Users pages

**Root cause:** Duplicate `document.addEventListener('click', ...)` handlers in page-specific `{% block scripts %}` fired *before* the global handler in `base.html`. Page handler opened the menu; base handler saw it open and closed it immediately.

**Secondary cause:** Players list wrapper used `<div style="overflow-x:auto">` which clips absolutely-positioned dropdowns even after the `open` class is added.

**Files changed:**
- `templates/players/detail.html` — removed duplicate dropdown handler from IIFE
- `templates/auth/users_list.html` — removed duplicate dropdown handler
- `templates/events/detail.html` — removed duplicate dropdown handler
- `static/js/players-table.js` — removed `closeAllDropdowns()` + `initActionDropdowns()` (~32 lines)
- `templates/players/list.html` — changed `<div style="overflow-x:auto">` → `<div class="table-responsive">` so `dropdown-open` mechanism works

**Commits:** `eb2856b`

## 2. Magic link logs in wrong user (event reminder emails)

**Root cause:** `services/scheduler.py` and `services/channels/email_channel.py` passed `player.id` to `create_magic_link()`, which expects a `user.id`. The two ID spaces are independent — when `player.id` happened to equal another user's `user.id`, clicking the magic link logged in as that other user.

**Example:** Zaccaria Rossetti (player.id=29) received an event reminder whose magic link encoded user.id=29 (which belonged to Rocco Canevascini), so clicking it logged him in as Rocco.

**Fix:** Use `player.user.id` (linked User FK) instead of `player.id`. Guard with `if player.user_id else None` to skip magic link when player has no linked user account.

**Files changed:**
- `services/scheduler.py` line 88
- `services/channels/email_channel.py` line 22

**Commits:** `8737b0e`

## 3. Magic link security hardening — email claim

**Problem:** Magic link tokens only encoded `user_id`. A token generated for the wrong user_id (bug #2 above) could not be detected at login time.

**Fix:** `create_magic_link(user_id, redirect_path, email)` now encodes the recipient's email as claim `"e"` in the token. `magic_link_login` verifies `token_email == user.email` and rejects mismatches. Backwards-compatible: legacy tokens without `"e"` still work.

**Files changed:**
- `services/auth_service.py` — `create_magic_link` + `verify_magic_link` signatures
- `routes/auth.py` — email mismatch check in `magic_link_login`
- `routes/users.py` — 4 callers updated with `email` arg
- `services/scheduler.py` — updated with `player.user.email`
- `services/channels/email_channel.py` — updated with `player.user.email`

**Commits:** `87a41c8`

## 4. Remove confirm popups on player detail user actions

**Problem:** Reset Password and Send Welcome in the player profile's User Account section showed a `confirm()` dialog and then redirected to `/auth/users` (instead of back to the player page).

**Fix:** Removed `onsubmit="return confirm(...)"` from both forms. Added hidden `<input name="next" value="/players/{{ player.id }}">`. Route reads `next` param and redirects to player profile with `?reset=1` / `?welcome=1` query params. Player detail page shows success banner on those params.

**Files changed:**
- `templates/players/detail.html` — removed `onsubmit`, added `next` hidden field + success banners
- `routes/users.py` — `reset_password` + `send_welcome` read `next` param

**Commits:** `f0d0206`, `af68618`

## 5. Telegram bot not starting (Docker bridge network issue)

**Root cause:** Docker containers on pi5 cannot make outbound TCP connections (ConnectTimeout to `api.telegram.org`). The host network works fine; Docker's bridge interface lacks outbound routing.

**Fix:** Added `network_mode: host` to `web` service in `docker-compose.override.yml` (pi5-local, gitignored). Container now uses host network stack and reaches Telegram API.

**Also fixed:** Same DNS/network issue was blocking `docker compose build` (pip install failed). Added `build: network: host` to override for both `web` and `backup` services.

**Files changed (pi5-local only):**
- `~/dockerimages/proManager/docker-compose.override.yml`
